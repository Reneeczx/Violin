## 1. Product Summary

`Violin` 是一个面向小提琴初学者及其家庭场景的练习 Web App。产品目标不是替代老师，而是在两次上课之间提供“每天该练什么、怎么练、如何自查”的轻量练习支架。

当前产品基于“一周一课”的节奏运作，通过静态课程配置生成每日练习内容，并在浏览器中提供参考音、节拍、示范播放、录音、周计划、进度统计和基础乐理解释。

## 2. Problem Statement

在课后练习阶段，初学者和家长通常面临以下问题：

- 不知道当天应该练哪几项，顺序和时长都不清楚。
- 看不懂最基本的乐理符号，例如 `4/4`、`♩=48`、`pizz.`、休止符。
- 没有稳定的参考音与节拍支持，练习容易“练偏”。
- 老师不在身边时，孩子缺少可重复、低门槛的自我检查方式。
- 家长缺少一个清晰、轻量的“今天练完了吗”的反馈界面。

## 3. Target Users

### Primary Users

- 小提琴儿童初学者
- 陪练家长

### Secondary Users

- 负责布置课后练习内容的小提琴老师

## 4. Product Goals

### Primary Goals

- 让学生在 1 分钟内明确“今天练什么”。
- 让学生能在没有老师陪同的情况下完成基础空弦和入门曲目练习。
- 让家长能快速确认当天练习是否完成。
- 让最常见的初学者乐理困惑在练习现场就被解释清楚。

### Success Definition

如果一个初学者在 iPad 或手机上打开应用后，能够直接完成以下动作，则说明产品达到了当前阶段目标：

- 进入首页并看到当日练习项
- 点击空弦按钮并听到参考音
- 跟随曲目高亮和示范完成一轮练习
- 打开基础乐理说明理解速度和拍号
- 勾选完成练习并在进度页看到结果

## 5. Non-Goals

当前阶段明确不做：

- 教师内容管理后台
- 多用户账号体系
- 云端数据同步
- AI 评音、自动纠错
- 多课程并行管理
- 社交、排行榜、社区

## 6. User Jobs To Be Done

### Student

- 当我开始练琴时，我想知道今天先练什么、后练什么，这样我不用自己决定。
- 当我看到乐谱和速度记号时，我想马上知道它们是什么意思，这样我不会因为看不懂而中断练习。
- 当我练空弦或小曲时，我想听到标准音和节拍，这样我能更容易跟上。

### Parent

- 当孩子开始练习时，我想有一个简单的每日清单，这样我不用临时回忆老师说了什么。
- 当孩子说“练完了”时，我想快速看到完成状态，这样我能判断练习是否落实。

### Teacher

- 当我布置一周练习时，我希望每天的重点能被清晰映射到产品里，这样学生在家里也能按节奏练。

## 7. Scope

### In Scope

- 基于当前课程数据自动生成 7 天练习计划
- 首页展示当日练习项、重点、建议速度、完成状态
- 空弦练习参考音
- 曲目练习谱面展示、每拍编号、示范播放、速度切换
- 节拍器
- 调弦助手
- 录音、回放、删除
- 本周计划
- 进度统计
- 基础乐理页面
- PWA 安装与基本离线缓存

### Out of Scope

- 后端服务
- 用户登录
- 云端录音保存
- AI 识别演奏质量
- 老师批改工作流

## 8. Functional Requirements

### FR-1 Daily Practice Plan

- 系统必须根据当前课程与日期生成当日练习计划。
- 每日练习计划必须包含主题、总时长和练习分项。

### FR-2 Open Strings Practice

- 系统必须支持 `E / A / D / G` 四根空弦参考音。
- 同一个参考音再次点击时必须停止。
- 点击不同弦时必须切换到新的参考音。
- 空弦练习必须支持节拍器和录音。

### FR-3 Piece Practice

- 系统必须展示按小节组织的谱面。
- 系统必须在谱面上方显示拍号编号。
- 系统必须支持示范播放。
- 系统必须支持速度切换。
- 示范播放与节拍器必须互斥。
- 曲目练习必须支持录音。

### FR-4 Tuner

- 系统必须提供四根空弦的调弦参考音。
- 系统必须支持“依次播放全部”。

### FR-5 Theory Help

- 系统必须对初学者高频困惑提供解释。
- 当前至少覆盖：
  - 拍号
  - 速度记号
  - 休止符
  - `pizz.` / `arco`
  - 空弦和音名

### FR-6 Progress Tracking

- 系统必须记录每日练习完成情况。
- 系统必须展示连续练习天数、累计练习时长和本周完成情况。

### FR-7 Recording

- 系统必须支持录音、回放和删除。
- 录音必须按练习项、日期和周次归档。

## 9. Non-Functional Requirements

- 移动端优先，iPad 可用
- 静态托管可运行，无后端依赖
- iPad / Safari 音频交互尽量稳定
- Service Worker 不应让旧脚本长期滞留
- 用词应适合初学者和家长理解

## 10. UX Principles

- 先做当天最需要的事，不让用户先理解系统结构
- 任何专业术语都尽量就地解释
- 对初学者来说，反馈应直观且低负担
- 界面优先服务练习流程，而不是展示复杂信息

## 11. Success Metrics

当前阶段用产品可用性指标，而非商业指标：

- 首页打开后 1 分钟内可进入练习
- 用户能直接听到参考音、节拍器和示范
- 初学者能在乐谱区域理解 `♩=xx`、`4/4`、休止符的含义
- 家长可以明确知道“今天练没练完”

## 12. Risks And Constraints

- 当前仅支持一份活动课程数据
- 本地存储不跨设备同步
- 真机音频行为受设备设置影响，例如静音模式和媒体音量
- 首页 view 文件目前仍偏大，后续维护成本需继续关注

## 13. Release Readiness Checklist

- 当前课程数据完整
- 当日练习卡片渲染正确
- 空弦参考音、示范播放、节拍器可用
- iPad / Safari 真机验证通过
- 基础乐理说明文案正确
- 变更已更新到 `README / CHANGELOG / VERSION`

## 14. Open Questions

- 后续是否需要支持多节课内容切换
- 是否需要老师可编辑课程数据的半自动工作流
- 是否要增加家长视角的“今日完成反馈摘要”

## 15. Learning Mode MVP Draft

### Goal

为不会具体编程、但希望通过项目学习工程设计的初学者，提供一个独立的可视化学习页，帮助其顺着真实项目理解：

- 应用从哪里开始
- 模块之间如何调用 / 依赖
- 当前主线里的数据如何流动
- 某个核心模块为什么要这样设计

### User

- Primary: 具备 CS 背景、但编程经验有限的项目拥有者
- Secondary: 需要快速理解项目结构的协作者

### Must-Have Content

- 模块关系图
  - 单页展示
  - 同时显示稳定依赖关系和当前主线高亮
  - 图是视觉中心，不应被说明卡挤压
- 关键概念
  - 只保留 8 到 10 个初学者必须掌握的概念
  - 概念必须和当前项目中的真实模块绑定，而不是做泛化术语表
- 核心代码解读
  - 每个核心模块只讲 1 到 2 段关键代码
  - 每段代码都要包含：文件位置、这段代码在做什么、为什么这样设计、建议继续追读到哪里

### Main Interaction Flow

- 用户进入独立学习页
- 默认看到一张模块关系图，以及默认主线
- 点击图中的模块后，页面同步更新：
  - 当前模块职责
  - 与该模块相关的关键概念
  - 与该模块相关的 1 到 2 段核心代码解读
- 切换主线后：
  - 图中的高亮线更新
  - 当前模块相关说明继续保持围绕所选模块

### UI Requirements

- 页面结构采用单页布局
- 顶部是简短引导，不应占用过多首屏空间
- 模块关系图位于页面上半部分，并优先获得桌面端宽度
- 图中的箭头和线条只服务于理解关系，不应抢过节点本身的视觉权重
- 概念区和代码解读区位于图下方，围绕当前模块展开，而不是独立堆很多卡片

### Constraints

- 学习页保持独立入口，不并入主练习页导航
- 不新增前端依赖，继续使用原生 HTML / CSS / JS
- 内容必须基于真实项目文件与真实模块
- 代码解读必须映射到真实源码，不接受纯概念化说明
- 页面要尽量降低认知负荷，避免一次展示过多概念或过多代码

### Deferred

- 学习任务 / 打卡系统
- 用户笔记、批注、收藏
- 完整术语词典
- 逐段或逐行代码导读
- 自动从源码抽取并高亮复杂代码片段

### Completion Criteria

如果一个初学者在不阅读长文档的前提下，能通过该页面回答以下问题，则认为 MVP 达标：

- 这个应用从哪里开始启动
- “今日练习生成”这条主线经过哪些模块
- 当前选中的模块依赖谁、又被谁使用
- 该模块最重要的 1 到 2 段代码分别在做什么
